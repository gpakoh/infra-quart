# Agent Orchestration Rules

Правила параллельной работы агентов в проектах `quart-platform`.

---

## 1. Агенты не видят друг друга

Каждый агент работает в изолированном окружении. Нет общего состояния, разделяемой памяти, IPC или RPC между агентами. Единственный способ обмена — через репозиторий.

## 2. Синхронизация только через Git/Gitea/GitHub

Все изменения передаются через коммиты, ветки, PR и теги. Никаких side-каналов:

- `git push` — отправить результат
- `git fetch` / `git pull` — получить результат другого агента
- Gitea API — проверить статус PR, CI, наличие тега

## 3. Ожидание зависимостей

Если задача агента зависит от результата другого агента:

```
1. Определить маркер готовности: commit SHA, tag, branch name, PR number
2. git fetch origin
3. Проверить наличие маркера локально
4. Если нет — sleep 120
5. Повторить шаги 2–4
```

Не делать предположений о времени выполнения другого агента.

## 4. Именование агентов

| Правильно | Неправильно |
|-----------|-------------|
| Agent 1, Agent 2, Agent 3 | Agent A, Agent B, Agent C |
| Agent 4, Agent 5 | Agent D, Agent E |

Цифровые идентификаторы исключают путаницу в логах и коммуникации.

## 5. Запрет push в protected main

Pull request — единственный способ внести изменения в `main`, если ветка защищена. Прямой push в `main` запрещён.

## 6. Запрет force-push без разрешения

`git push --force` только после явного одобрения в задаче. Force-push на shared ветке — авария.

## 7. Submodule bump

Обновление submodule (`git submodule update --remote` / изменение `.gitmodules` / `.gitattributes`) выполняется **только после**:

- merge соответствующего PR в `main`, **или**
- появления стабильного tag'а

Submodule bump не делается «на глаз» или по незавершённой ветке.

## 8. CI failures unrelated to scope

Если CI падает по известной проблеме, не связанной с текущей задачей (например, `coverage.xml` баг в другом компоненте):

- зафиксировать в комментарии к задаче
- не чинить в чужом scope
- продолжить работу

## 9. Один агент — одна ветка

Каждый агент работает в своей ветке. Никакой работы в чужой ветке. Никаких коммитов поверх ветки другого агента.

## 10. Ожидание вместо вмешательства

Если нет независимой задачи — агент ждёт. Не лезть в чужой scope, не править чужой код, не решать чужие проблемы без явного указания.
